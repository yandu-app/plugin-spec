# Plugin Internationalization

Plugins can provide their own translations. Yandu merges them into the runtime i18n system automatically.

## Locale File Convention

Place locale files in the plugin package at:

```
yandu-plugin-hello/
├── package.json
├── src/
│   └── index.ts
└── locales/
    ├── en.json
    └── zh.json
```

## Locale File Format

```json
{
  "name": "Hello Plugin",
  "description": "A simple greeting plugin",
  "settings": {
    "greeting": {
      "label": "Greeting Text",
      "hint": "The text shown when greeting"
    }
  }
}
```

## Namespace

Each plugin gets its own i18next namespace equal to its package name (e.g. `yandu-plugin-hello`).

In the plugin's config schema, use the namespace-prefixed key:

```typescript
import type { ConfigSchema } from '@yandu/types';

const schema: ConfigSchema = {
  id: 'hello',
  sections: [{
    fields: [{
      key: 'hello.greeting',
      type: 'text',
      hintI18nKey: 'yandu-plugin-hello:settings.greeting.hint',
      defaultValue: 'Hello',
    }],
  }],
};
```

The `hintI18nKey` format is `{namespace}:{key}`. Yandu resolves this at render time.

## Programmatic Access

Plugins can also register locales programmatically via `KernelSystem`:

```typescript
system.i18n.addResourceBundle('yandu-plugin-hello', 'en', {
  name: 'Hello Plugin',
  greeting: 'Hello, {name}!',
});
```

This is useful when locales need to be computed at runtime.

## Supported Languages

Yandu currently supports:
- `en` — English
- `zh` — Chinese (Simplified)

Plugins should provide at least `en.json`. `zh.json` is optional but recommended.

## Fallback

If a key is missing in the user's language, Yandu falls back to `en`. If `en` is also missing, the key itself is displayed.
